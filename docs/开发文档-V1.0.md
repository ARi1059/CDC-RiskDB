# Telegram 黑名单共享机器人开发文档 V1.0

## 项目简介

本项目为 Telegram 黑名单共享机器人。

机器人主要用于老师之间共享黑名单数据，实现用户查询、黑名单录入、黑名单管理等功能。

机器人采用全键盘按钮交互模式，不使用命令操作，不要求用户手动输入 Telegram 用户名或数字 ID。

机器人通过 Telegram 用户分享功能（Request User）获取目标用户信息，解决 Forward Privacy 导致无法获取用户 ID 的问题。

## 重复录入处理

为保证「N 条记录 = N 位老师佐证」的可信度，**同一位老师对同一目标用户只能保留 1 条有效记录**。

> 唯一性范围：`(operator_id, telegram_id)`，且仅约束 **有效记录**（软删除的行不占名额）。**不同老师**录入同一用户不受限制 —— 这是共享佐证的核心特性。

### 检测时机

老师在「🚫 录入黑名单 → 👤 选择用户」选定目标后，机器人立即检测该 `(录入人, 目标用户)` 是否已存在 **有效记录**：

- **无有效记录**（从未录入，或此前记录已被删除）→ 正常进入「选择拉黑原因」流程；删除后重新录入视为新增，会照常触发广播。
- **已有有效记录** → 不新增记录，进入下方「重复提示」分支。

### 重复提示

```text
━━━━━━━━━━━━━━

⚠️ 您已录入过该用户

目标用户：
@username

数字ID：
123456789

原有原因：
诈骗

录入时间：
2026-06-25 18:30

━━━━━━━━━━━━━━

如需修改，可更新原因或删除记录
```

按钮：

```text
🔄 更新原因

🗑 删除记录

🏠 返回首页
```

### 行为说明

- **🔄 更新原因**：重新选择拉黑原因 → 覆盖原记录 `reason`、刷新 `updated_at`；**不新增记录、不触发广播**。
- **🗑 删除记录**：**软删除** 该老师对此用户的记录（置 `deleted_at`，不物理删除）。删除后可重新录入。
- **🏠 返回首页**：不做任何改动。
- 重复录入与更新原因 **全程不触发**「有新黑名单录入，请查看」广播（非新增拉黑）。

### 更新成功

```text
✅ 原因已更新

目标用户：
@username

新原因：
跑单

更新时间：
2026-06-25 18:45
```

按钮：

```text
🔍 查询用户

🏠 返回首页
```

### 并发兜底

数据库对 `(operator_id, telegram_id)` 建 **部分唯一索引**（仅约束 `deleted_at IS NULL` 的有效行）；即使应用层检测被连点 / 并发绕过，第二条有效记录也会因唯一冲突被拒，不会产生重复有效行。

## 录入广播通知

每当有新的黑名单录入成功后，机器人自动向其他老师私聊推送一条提醒，告知有新记录产生，引导其主动查询。

### 触发时机

老师点击 ✅ 确认录入、记录成功写入共享黑名单后立即触发；广播为异步执行，不阻塞录入人的「✅ 录入成功」回执。

### 接收对象

- ✅ 全部 **在职 Teacher（老师）** 角色用户（`is_active = true`）
- ❌ **不通知管理员（Admin）**
- ❌ **不通知已停用的老师**
- ❌ **不通知本次录入人本人**

> 即：收件人 = 全部在职老师 − 录入人；管理员与已停用老师不接收。

### 推送方式

机器人逐个 **私聊推送**（`sendMessage`）给每位接收人。

### 消息内容

老师名称以 **超链接** 形式指向其用户名，点击即可跳转该老师的 Telegram 主页：

```text
张老师 有新的黑名单录入，请查看
```

> 其中「张老师」为可点击的蓝色超链接，指向录入人的用户名主页。

#### 消息格式（HTML parse_mode）

```html
<a href="https://t.me/teacher001">张老师</a> 有新的黑名单录入，请查看
```

- **老师名称**：取录入人的 `first_name`（发送前需做 HTML 转义；如需也可改为显示 `@username`）
- **超链接目标**：`https://t.me/<录入人 username>`
- **无 username 时降级**：使用 `tg://user?id=<录入人 telegram_id>` 作为链接目标（内联提及），仍可点击跳转

> 通知内 **不包含被拉黑用户的具体信息**（用户名 / ID / 原因）。老师收到提醒后，使用主菜单 🔍 查询用户 按钮自行查询，遵循「全键盘按钮、不手输 ID」原则。

### 实现要点

1. **异步广播**：录入成功后 fire-and-forget 或入队执行，避免阻塞主流程与录入人回执。
2. **逐人容错**：对每位接收人单独 `try/catch`；若对方未启动机器人或已拉黑机器人（`403 Forbidden`），跳过并继续，不影响其他人。
3. **限流**：Telegram 群发约 30 条/秒上限，老师较多时需节流（建议 ≤ 25 条/秒）分批发送。
4. **解析模式**：建议使用 `HTML` parse_mode，并对 `first_name` 做转义，避免特殊字符破坏排版。

# 数据库设计

## users

```sql
CREATE TABLE users (
    telegram_id BIGINT PRIMARY KEY,
    username VARCHAR(255),
    first_name VARCHAR(255),
    role ENUM('admin','teacher'),
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

> **`is_active`**：是否在职。移除老师为 **停用**（置 `false`），保留用户行；授权门禁仅放行 `is_active = true` 的用户。
>
> 支持 **多名管理员**：`role = 'admin'` 可有多行，系统须至少保留 1 名。

---

## blacklist

```sql
CREATE TABLE blacklist (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    telegram_id BIGINT NOT NULL,

    username VARCHAR(255),

    first_name VARCHAR(255),

    reason VARCHAR(100),

    operator_id BIGINT NOT NULL,

    operator_username VARCHAR(255),

    created_at TIMESTAMP,

    updated_at TIMESTAMP,

    deleted_at TIMESTAMP NULL          -- 软删除：NULL = 有效，非空 = 已删除
);

-- 同一老师对同一用户仅一条「有效」记录；软删除行不占用名额，允许删除后重新录入
CREATE UNIQUE INDEX blacklist_operator_target_active_uniq
    ON blacklist (operator_id, telegram_id)
    WHERE deleted_at IS NULL;
```

> **部分唯一索引（`... WHERE deleted_at IS NULL`）**：保证同一老师对同一用户仅一条 **有效** 记录；软删除的行不占名额，因此删除后可重新录入。不同老师录入同一用户不受影响。
>
> **`deleted_at`**：软删除标记（NULL = 有效）。删除记录仅置位、不物理删除，便于审计与恢复；所有面向用户的查询均过滤 `deleted_at IS NULL`。
>
> **`updated_at`**：最后一次「更新原因」时间；`created_at` 保留首次录入时间。

# 开发原则

- 全键盘按钮交互
- 不使用命令操作
- 不允许手动输入 Telegram ID
- 不允许手动输入用户名
- 使用 Telegram User Share 获取用户信息
- 黑名单录入后立即生效
- 不设置审核流程
- 仅保留 Admin 与 Teacher 两种角色（支持多名管理员，至少保留 1 名）
- 所有黑名单数据全局共享
- 新黑名单录入后自动私聊广播提醒其他老师（不含管理员、已停用老师与录入人）
- 同一老师对同一用户仅保留一条 **有效** 记录（可更新原因 / 删除后可重录），不同老师可分别录入
- 黑名单删除采用 **软删除**（标记 `deleted_at`，保留可追溯，不物理删除）
- 移除老师为 **停用功能**（`is_active = false`），其历史录入记录全部保留并继续生效
