## 一、技术选型（已锁定）

| 项 | 选型 | 说明 |
|---|---|---|
| 语言 | **TypeScript** | 类型安全，配合 Telegraf 与 Prisma 类型 |
| Bot 框架 | **Telegraf**（≥ 4.15） | 需支持 `KeyboardButtonRequestUsers` / `users_shared` |

| 配置 / 校验 | **zod** | env 与输入校验 |
| 日志 | **pino** | 结构化日志 |

## 三、数据模型（Prisma schema 草图）

对应需求文档 [数据库设计](开发文档-V1.0.md)，并补充查询所需索引与唯一约束。

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

enum Role {
  admin
  teacher
}

model User {
  telegramId BigInt   @id @map("telegram_id")
  username   String?  @db.VarChar(255)
  firstName  String?  @map("first_name") @db.VarChar(255)
  role       Role
  isActive   Boolean  @default(true) @map("is_active")   // 移除老师=停用；门禁仅放行 true
  createdAt  DateTime @default(now()) @map("created_at")
  updatedAt  DateTime @updatedAt       @map("updated_at")

  @@map("users")
}

model Blacklist {
  id               BigInt   @id @default(autoincrement())
  telegramId       BigInt   @map("telegram_id")
  username         String?  @db.VarChar(255)
  firstName        String?  @map("first_name") @db.VarChar(255)
  reason           String   @db.VarChar(100)
  operatorId       BigInt   @map("operator_id")
  operatorUsername String?  @map("operator_username") @db.VarChar(255)
  createdAt        DateTime @default(now()) @map("created_at")
  updatedAt        DateTime @updatedAt       @map("updated_at")
  deletedAt        DateTime? @map("deleted_at")   // 软删除：NULL=有效

  // Prisma 不支持「部分唯一索引」，唯一性改用迁移 SQL 手写（见 schema 块下方说明）
  @@index([telegramId])                // 查询用户：按目标聚合
  @@index([operatorId])                // 判重 / 按录入人聚合
  @@map("blacklist")
}

model Announcement {
  id        BigInt   @id @default(autoincrement())
  content   String   @db.Text
  createdBy BigInt   @map("created_by")
  createdAt DateTime @default(now()) @map("created_at")

  @@map("announcements")
}
```

> **部分唯一索引（重要）**：`(operator_id, telegram_id)` 的唯一性仅约束有效行（`WHERE deleted_at IS NULL`），软删除后可重新录入。Prisma 无法在 schema 中声明部分唯一索引 —— 用 `prisma migrate dev --create-only` 生成迁移后，在 SQL 末尾追加：
>
> ```sql
> CREATE UNIQUE INDEX blacklist_operator_target_active_uniq
>   ON "blacklist" (operator_id, telegram_id) WHERE deleted_at IS NULL;
> ```
>
> 再 `prisma migrate dev` 应用。所有面向用户的查询都要带 `deletedAt: null` 过滤。

## 四、开发里程碑

### 里程碑总览

| 里程碑 | 名称 | 完成标准（DoD） | 依赖 | 规模 |
|---|---|---|---|---|
| **M0** | 脚手架与基础设施 | 本地 `npm run dev` 起空 bot，连通 PG+Redis，回显 ok | — | M |
| **M1** | 数据模型与迁移 | 三表 + 部分唯一索引迁移落地，初始管理员（可多名）引导成功 | M0 | S |
| **M2** | 鉴权·角色·主菜单 | 授权用户见角色菜单，未授权被拒 | M1 | M |
| **M3** | 用户分享基建 | 任一「选择用户」稳定取 id / username / first_name | M2 | M |
| **M4** | 查询用户 | 选用户 → 有 / 无记录结果 | M3 | S |
| **M5** | 录入黑名单 | 选用户 → 判重 → 选原因 → 确认 → 成功；含更新原因 | M3·M4 | L |
| **M6** | 录入广播通知 | 异步私聊广播（在职老师，除录入人、不含 admin） | M5 | M |
| **M8** | 公告 + 管理员管理 | Teacher 查看公告；Admin 发布/管理公告、管理管理员 | M2 | M |
| **M9** | 老师管理 | 添加 / 停用 / 列表（停用保留记录） | M3 | M |
| **M10** | 加固·测试·上线 | systemd 托管，部署 Ubuntu 24.04 | M0–M9 | L |

> 规模为相对工作量（S < M < L），非工时承诺。

### M0 — 脚手架与基础设施

- 初始化 npm + TypeScript（tsconfig、eslint、prettier）
- 安装 telegraf / prisma / @prisma/client / ioredis / zod / dotenv / pino
- `env.ts`：zod 校验 `BOT_TOKEN` / `DATABASE_URL` / `REDIS_URL` / `ADMIN_TELEGRAM_IDS`
- `prisma.ts`、`redis/client.ts` 单例连接
- `bot.ts`：Telegraf 实例 + long polling + 优雅退出（SIGINT/SIGTERM）+ `bot.catch`
- 本地依赖：PostgreSQL 16 + Redis 7（Homebrew / apt 安装或既有实例），连接串写入 `.env`
- **产出**：空 bot 可启动，对任意消息回 `ok`。

### M1 — 数据模型与迁移

- `schema.prisma`：User（含 `is_active`）/ Blacklist（含 `deleted_at`）/ Announcement + `Role` 枚举（见上）
- `prisma migrate dev --create-only` 生成迁移后，手写追加 **部分唯一索引**（`WHERE deleted_at IS NULL`），再应用
- 启动引导脚本：解析 `ADMIN_TELEGRAM_IDS`（逗号分隔），逐个 upsert 为 `admin` + `is_active=true`
- **产出**：三表 + 部分唯一索引 + 普通索引就绪；初始管理员（可多名）可被识别。

### M2 — 鉴权·角色·主菜单框架

- `auth` 中间件：由 `ctx.from` 查 users → 注入 `ctx.state.user / role`
- 白名单门禁：不在 users 表、或 `is_active=false` 的用户提示「无权限」并拦截
- 主菜单键盘（Teacher / Admin），统一「🏠 返回首页」
- Redis session 中间件（`@telegraf/session` + Redis store）
- **产出**：不同角色登录见对应菜单，越权被拦。

### M3 — 用户分享基建（KeyboardButtonRequestUsers）

- 封装 `requestUser` 键盘，用 `request_id` 路由场景（查询 / 录入 / 加老师）
- 开启 `request_username` / `request_name`，统一处理 `users_shared` 更新
- 无 username 的降级处理（见关键技术点）
- **产出**：三处复用的「选择用户」底座，真机验证可取到字段。

### M4 — 查询用户

- `blacklist.service.findByTelegramId` 聚合多条记录
- 渲染「查询结果」：有记录（状态徽标 + 原因 + 录入老师列表）/ 无记录
- **产出**：[查询用户](开发文档-V1.0.md) 流程闭环。

### M5 — 录入黑名单（核心）

- 选用户后判重：查 `(operatorId, telegramId)` 的 **有效** 记录（`deletedAt: null`）
- 无有效记录 → 原因键盘 → 确认面板 → 写库（删除后重录＝插入新行）→ 成功面板 + 触发广播
- 有有效记录 → 重复提示面板 → 🔄 更新原因 / 🗑 删除（软删）/ 🏠 返回
- 更新原因 → 覆盖 `reason` + `updatedAt`（不广播）→ 已更新面板
- 部分唯一索引冲突（Prisma `P2002`）兜底；确认按钮幂等防连点
- **产出**：[录入黑名单](开发文档-V1.0.md) + [重复录入处理](开发文档-V1.0.md) 全流程。

### M6 — 录入广播通知

- `broadcast.service`：收件人 = `role=teacher` 且 `is_active=true` 且 `id ≠ operatorId`（不含 admin、不含已停用老师）
- HTML 超链接：`first_name` → `https://t.me/<username>`；无 username 降级 `tg://user?id=`
- 逐人 try/catch，`403` 跳过；限流 ≤ 25 条/秒；fire-and-forget
- **产出**：[录入广播通知](开发文档-V1.0.md) 闭环，不阻塞录入回执。

### M8 — 机器人公告（查看）+ 系统设置（公告 + 管理员管理）

- `announcement.service`：create / latest / list / delete
- Teacher：查看最新公告
- Admin：发布公告（输入正文）、公告管理（列表 / 删除）
- **管理员管理**：添加管理员（选用户 → `role=admin`）/ 移除管理员（降级为 `teacher`）/ 管理员列表；**护栏：不能移除最后一名管理员**
- **产出**：[机器人公告](开发文档-V1.0.md) + [系统设置](开发文档-V1.0.md)（含管理员管理）。

### M9 — 老师管理（Admin）

- 添加老师：选用户 → upsert users `role=teacher` + `is_active=true`（已停用者即恢复）→ 授权成功
- 移除老师：在职列表 → 选择 → 确认 → **停用**（`is_active=false`，保留用户行与全部录入记录）
- 老师列表：仅在职（`is_active=true`）
- **产出**：[老师管理](开发文档-V1.0.md) 流程。

### M10 — 加固·测试·上线

- 边界与兜底：未知操作、状态过期、超时、错误文案
- 单元 + 关键流程集成测试（见测试策略）
- 结构化日志、健康检查
- 进程托管（systemd 服务 / PM2）、env 管理、PostgreSQL 定时备份、部署文档
- **产出**：可部署到 Ubuntu 24.04 的生产进程（systemd 托管）。

## 五、关键技术点

1. **用户分享（核心基建，M3）**
   - 依赖 Bot API ≥ 7.0 与较新 Telegraf：`KeyboardButtonRequestUsers` 开启 `request_username` / `request_name` 后，`users_shared` 返回 `user_id` + `username` + `first_name`。
   - 用 `request_id` 区分查询 / 录入 / 加老师三个场景。
   - **隐私降级**：用户未设公开 username 时仍可能取不到 → 录入与广播链接改用 `tg://user?id=<id>`。
   - M3 必须真机验证字段可达，否则后续全流程受阻。

2. **多步流程与状态（Scenes + Redis）**
   - 「选用户 → 选原因 → 确认」等用 Telegraf Scenes / WizardScene。
   - 会话状态存 Redis：重启不丢、可水平扩展。

3. **软删除、判重与幂等**
   - 判重只看 **有效** 记录（`deletedAt: null`）；软删除的行不阻止重新录入。
   - 唯一性用 **部分唯一索引**（`WHERE deleted_at IS NULL`）做并发兜底，捕获 `P2002`；删除后重录插入新行、保留历史。
   - 所有面向用户的查询统一加 `deletedAt: null` 过滤（建议封装到 service 层，避免遗漏）。
   - 确认后立即移除/禁用键盘，防连点双提交。

4. **广播限流与容错**
   - `p-limit` 或令牌桶限速 ≤ 25 条/秒；逐人 try/catch，`403`（被拉黑 / 未启动）跳过并计数。
   - fire-and-forget，异常进日志，不影响录入人「✅ 录入成功」回执。

5. **消息排版与转义**
   - 统一 `HTML` parse_mode；`first_name` / `username` 发送前做 HTML 转义。
   - 按钮文案、原因、模板集中常量管理，避免散落。

6. **角色门禁与多管理员**
   - 中间件统一鉴权：仅 `is_active=true` 用户放行；admin-only 场景（老师管理 / 系统设置 / 管理员管理）入口二次校验。
   - 支持多名 `admin`；移除管理员前校验「至少保留 1 名」，避免锁死。

## 八、部署

- **备份**：`pg_dump` 定时备份。
- **时区**：`TZ=Asia/Shanghai`，保证录入时间显示一致。

## 十、已定决策与待定项

### 已定（已回填需求文档）

- **删除策略 = 软删除**：`blacklist.deleted_at` 标记、物理保留；唯一性改部分唯一索引，删除后可重录。
- **移除老师 = 停用**：`users.is_active = false`，历史录入记录全部保留并继续生效。
- **支持多管理员**：`role = 'admin'` 可多名，经 `ADMIN_TELEGRAM_IDS` 引导 +「管理员管理」增减，至少保留 1 名。

### 待定

1. **公告正文输入**：发布公告是唯一需手输文本的场景；与「不手输 Telegram ID / 用户名」原则 **不冲突**（输入的是公告正文，非用户标识）。需定多行 / 取消交互。
2. **用户分享字段可达性**：依赖 Bot API ≥ 7.0 与较新 Telegraf，M3 真机验证为前置门槛。
3. **删除记录是否二次确认**：需求文档未要求；建议加一步确认防误删，待定。
4. **「已删除 / 已停用」审计视图**：软删除与停用都保留了数据，是否在 Admin 侧提供查看 / 恢复入口（V1 可不做）。
